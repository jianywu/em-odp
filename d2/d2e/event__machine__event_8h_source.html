36 /**
37  * @file
38  * @defgroup em_event Events
39  * Operations on an event.
40  * @{
41  *
42  * All application processing is driven by events in the Event Machine. An event
43  * describes a piece of work or data to be processed. The structure of an event
44  * is implementation and event type specific: it may be a directly accessible
45  * buffer of memory, a descriptor containing a list of buffer pointers,
46  * a descriptor of a packet buffer etc.
47  *
48  * Applications use the event type to interpret the event structure.
49  *
50  * Events follow message passing semantics: an event has to be allocated using
51  * the provided API (em_alloc()) or received through queues by an EO callback
52  * function after which the event is owned by the application. Event ownership
53  * is transferred back to the system by using em_send() or em_free().
54  * An event not owned by the application should not be touched.
55  *
56  * Since em_event_t may not carry a direct pointer value to the event structure,
57  * em_event_pointer() must be used to translate an event to an event structure
58  * pointer (for maintaining portability).
59  *
60  * Additionally, an event may contain a user area separate from the event
61  * payload. The size of the event user area is set when creating the event pool
62  * from which the event is allocated. The user area is a fixed size (per pool)
63  * data area into which event related state data can be stored without having
64  * to access and change the payload. Note that the size of the event user area
65  * can be zero(0), depending on event pool configuration.
66  * Note that the user area content is not initialized by EM, neither em_alloc()
67  * nor em_free() will touch it and thus it might contain old user data set the
68  * last time the area was used during a previous allocation of the same event.
69  * Since the user area is not part of the event payload, it will not be
70  * transmitted as part of a packet etc.
71  * A user area ID can further be used to identify the user area contents.
72  * The event user area ID is stored outside of the user area itself and is thus
73  * always available, even if the size of the user area data is set to zero(0).
74  * See em_pool_create(), em_event_uarea_get(), em_event_uarea_id_get/set() and
75  * em_event_uarea_info() for more information on the event user area and its
76  * associated ID.
77  *
78  * em_event_t is defined in event_machine_types.h
79  *
80  * @see em_event_pointer()
81  */

90 /**
91  * Allocate an event.
92  *
93  * The memory address of the allocated event is system specific and can depend
94  * on the given pool, event size and type. The returned event (handle) may refer
95  * to a memory buffer or a HW specific descriptor, i.e. the event structure is
96  * system specific.
97  *
98  * Use em_event_pointer() to convert an event (handle) to a pointer to the
99  * event payload. EM does not initialize the payload data.
100  *
101  * EM_EVENT_TYPE_SW with minor type '0' is reserved for direct portability -
102  * it is always guaranteed to produce an event with contiguous payload that can
103  * directly be used by the application up to the given size (no HW specific
104  * descriptors etc. are visible). This event payload will be 64-bit aligned
105  * by default (unless explicitly configured otherwise).
106  *
107  * EM_POOL_DEFAULT can be used as a pool handle if there's no need to use a
108  * specific event pool (up to the size- or event limits of that pool).
109  *
110  * Additionally it is guaranteed, that two separate buffers never share a cache
111  * line (to avoid false sharing).
112  *
113  * @param size Event size in octets (size > 0)
114  * @param type Event type to allocate
115  * @param pool Event pool handle
116  *
117  * @return The allocated event or EM_EVENT_UNDEF on error.
118  *
119  * @see em_free(), em_send(), em_event_pointer(), em_receive_func_t(),
120  * em_event_clone()
121  */
122 em_event_t em_alloc(size_t size, em_event_type_t type, em_pool_t pool);
123 
124 /**
125  * Allocate multiple events.
126  *
127  * Similar to em_alloc(), but allows allocation of multiple events, with same
128  * properties, with one function call.
129  * The em_alloc_multi() API function will try to allocate the requested number
130  * ('num') of events but may fail to do so, e.g. if the pool has run out of
131  * events, and will return the actual number of events that were successfully
132  * allocated from the given pool.
133  *
134  * @param[out] events Output event array, events are allocated and filled by
135  * em_alloc_multi(). The given array must fit 'num' events.
136  * @param num Number of events to allocate and write into 'events[]'
137  * @param size Event size in octets (size > 0)
138  * @param type Event type to allocate
139  * @param pool Event pool handle
140  *
141  * @return Number of events actually allocated from the pool (0 ... num) and
142  * written into the output array 'events[]'.

177 /**
178  * Send an event to a queue.
179  *
180  * The event must have been allocated with em_alloc(), or received via an EO
181  * receive-function. The sender must not touch the event after calling em_send()
182  * as the ownership has been transferred to the system or possibly to the next
183  * receiver. If the return status is *not* EM_OK, the ownership has not been
184  * transferred and the application is still responsible for the event (e.g. may
185  * free it).
186  *
187  * EM does not currently define guaranteed event delivery, i.e. EM_OK return
188  * value only means the event was accepted for delivery. It could still be lost
189  * during delivery (e.g. due to a removed queue or system congestion, etc).
190  *
191  * @param event Event to be sent
192  * @param queue Destination queue
193  *
194  * @return EM_OK if successful (accepted for delivery).
195  *
196  * @see em_alloc()
197  */
198 em_status_t em_send(em_event_t event, em_queue_t queue);

340 /**
341  * Mark the event as "sent".
342  *
343  * Indicates a user-given promise to EM that the event will later appear into
344  * 'queue' by some means other than an explicit user call to em_send...().
345  * Calling em_event_mark_send() transfers event ownership away from the user,
346  * and thus the event must not be used or touched by the user anymore (the only
347  * exception is (hw) error recovery where the "sent" state can be cancelled by
348  * using em_event_unmark_send() - dangerous!).
349  *
350  * Example use case:
351  * A user provided output-callback function associated with a queue of type
352  * 'EM_QUEUE_TYPE_OUTPUT' can use this API when configuring a HW-device to
353  * deliver the event back into EM. The HW will eventually "send" the event and
354  * it will "somehow" again appear into EM for the user to process.
355  *
356  * EM will, after this API-call, treat the event as "sent" and any further API
357  * operations or usage might lead to EM errors (depending on the error-check
358  * level), e.g. em_send/free/tmo_set/ack(event) etc. is forbidden after
359  * em_event_mark_send(event).
360  *
361  * @note Registered API-callback hooks for em_send...() (em_api_hook_send_t)
362  * will NOT be called.
363  * @note Marking an event "sent" with an event group (corresponding to
364  * em_send_group()) is currrently NOT supported.
365  *
366  * @param event Event to be marked as "sent"
367  * @param queue Destination queue (must be scheduled, i.e. atomic,
368  * parallel or parallel-ordered)
369  *
370  * @return EM_OK if successful
371  *
372  * @see em_send(), em_event_unmark_send()
373  */
374 em_status_t em_event_mark_send(em_event_t event, em_queue_t queue);
375 
376 /**
377  * Unmark an event previously marked as "sent" (i.e mark as "unsent")
378  *
379  * @note This is for recovery situations only and can potenially crash the
380  * application if used incorrectly!
381  *
382  * Revert an event's "sent" state, as set by em_event_mark_send(), back to the
383  * state before the mark-send function call.
384  * Any further usage of the event after em_event_mark_send(), by EM or
385  * the user, will result in error when calling em_event_unmark_send() since the
386  * state has become unrecoverable.
387  * => the only allowed EM API call after em_event_mark_send() is
388  * em_event_unmark_send() if it is certain that the event, due to some
389  * external error, will never be sent into EM again otherwise.
390  * Calling em_event_unmark_send() transfers event ownership back to the user
391  * again.
392  *
393  * @note Unmark-send and unmark-free are the only valid cases of using an event
394  * that the user no longer owns - all other such uses leads to fatal error
395  *
396  * @code
397  * em_status_t err;
398  * hw_err_t hw_err;
399  *
400  * // 'event' owned by the user
401  * err = em_event_mark_send(event, queue);
402  * if (err != EM_OK)
403  * return err; // NOK
404  * // 'event' no longer owned by the user - don't touch!
405  *
406  * hw_err = config_hw_to_send_event(...hw-cfg..., event, queue);
407  * if (hw_err) {
408  * // hw config error - the event can be recovered if it is
409  * // certain that the hw won't send that same event.
410  * // note: the user doesn't own the event here and actually
411  * // uses an obsolete event handle to recover the event.
412  * err = em_event_unmark_send(event);
413  * if (err != EM_OK)
414  * return err; // NOK
415  * // 'event' recovered, again owned by the user
416  * em_free(event);
417  * }
418  * @endcode
419  *
420  * @param event Event previously marked as "sent" with em_event_mark_send(),
421  * any other case will be invalid!
422  *
423  * @return EM_OK if successful
424  *
425  * @see em_send(), em_event_mark_send()
426  */
427 em_status_t em_event_unmark_send(em_event_t event);

557 /**
558  * @brief Get a pointer to the event user area, optionally along with its size.
559  *
560  * The event user area is a fixed sized area located within the event metadata
561  * (i.e. outside of the event payload) that can be used to store application
562  * specific event related data without the need to adjust the payload.
563  * The event user area is configured during EM event pool creation and thus the
564  * size of the user area is set per pool.
565  *
566  * Note that the user area content is not initialized by EM, neither em_alloc()
567  * nor em_free() will touch it and thus it might contain old user data set the
568  * last time the area was used during a previous allocation of the same event.
569  * Since the user area is not part of the event payload, it will not be
570  * transmitted as part of a packet etc.
571  *
572  * @param event Event handle to get the user area of
573  * @param[out] size Optional output arg into which the user area size is
574  * stored. Use 'size=NULL' if no size information is needed.
575  *
576  * @return a pointer to the event user area
577  * @retval NULL on error or if the event contains no user area
578  *
579  * @see em_pool_create() for pool specific configuration and
580  * the EM runtime config file em-odp.conf for the default value:
581  * 'pool.user_area_size'.

672 /**

677  * output via the 'uarea_info' struct.
678  * This API function combines the functionality of em_event_uarea_get() and
679  * em_event_uarea_id_get() for use cases where both the user area pointer as
680  * well as the ID is needed. Calling one API function instead of two might be
681  * faster due to a fewer checks and internal conversions.
682  *
683  * The event user area is a fixed sized area located within the event metadata
684  * (i.e. outside of the event payload) that can be used to store application
685  * specific event related data without the need to adjust the payload.
686  * The event user area is configured during EM event pool creation and thus the
687  * size of the user area is set per pool.
688  *
689  * Note that the user area content is not initialized by EM, neither em_alloc()
690  * nor em_free() will touch it and thus it might contain old user data set the

693  * transmitted as part of a packet etc.

696  * identify the contents of the actual user area data. The ID is stored
697  * outside of the actual user area data and is available for use even if the

699  * from. EM will initialize 'uarea_info.id.isset = false' when allocating
700  * a new event (indicating that the ID is not set).

703  * @param[out] uarea_info Output struct into which the user area information
704  * is stored.
705  *
706  * @return EM status code incidating success or failure of the operation.
707  * @retval EM_OK Operation successful.
708  * @retval Other Operation FAILED and no valid user area info could
709  * be obtained, 'uarea_info' is all NULL/zero(0) in this case.
710  *
711  * @see em_pool_create() for pool specific configuration and
712  * the EM runtime config file em-odp.conf for the default value:
713  * 'pool.user_area_size'.
714  * @see em_event_uarea_get(), em_event_uarea_id_get()
715  */
